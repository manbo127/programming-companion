3.需求与约束分析

3.1 功能性需求

本系统面向编程初学者，核心定位是一位耐心温和的智能学习陪伴者。其功能需求分为六项核心能力与四项扩展能力。

3.1.1 基础对话能力

学伴能与用户进行自然流畅的自由对话，自动识别用户意图所属的主题范围，包括代码错误求助、解题思路请求、知识点疑问以及一般性交流。学伴的角色定位为编程学习陪伴者，语气应友好、鼓励、耐心，让用户感受到被理解和支持。

3.1.2 代码错误解读能力

用户可以提交出错的代码片段以及系统返回的错误信息。学伴需要理解代码的意图，分析错误产生的原因，并用通俗易懂的语言向用户解释。此项能力的关键约束在于避免直接输出大段正确代码。回复应侧重于解释为什么出错以及应该往哪个方向修改，引导用户自己动手改正，而非代劳。

3.1.3 解题思路引导能力

当用户询问编程题目时，学伴不得直接给出完整答案代码，而应采用苏格拉底式的分步引导方法。首先帮助用户理解题目的输入和输出要求，然后提示解决问题的关键步骤，接着针对每个步骤提出引导性问题，最后根据用户的回答决定下一步的提示方向。整个过程通过多轮对话逐步推进，直到用户真正理解解题思路。

3.1.4 激励话术能力

学伴需要感知用户的学习状态并给予情感化反馈。主要识别两种典型场景：当用户连续正确或取得积极进展时，给予真诚的肯定和表扬；当用户连续出错或表现出挫败情绪时，给予温暖的鼓励和安慰。话术应自然真诚，避免机械重复，让用户感受到真实的陪伴而非套路化的应答。

3.1.5 交互界面

系统采用 Web 聊天界面的形式呈现。基本交互元素包括对话消息展示区、用户文本输入框和发送按钮。此外需要提供专用的代码输入区和错误信息输入区，方便用户结构化地提交信息。同时支持在统一输入框中按约定格式提交代码和错误信息，保持使用的灵活性。

3.1.6 对话上下文记忆能力

学伴需要记住本次对话过程中的完整交流历史，在后续回答中能够引用之前讨论过的内容。此外需要理解用户的代词指代，例如当用户说那个和之前有什么区别时，学伴应能正确识别那个指向的是上文提到的具体概念，确保多轮对话的自然连贯。

3.1.7 扩展功能

以下四项功能已纳入规划但本次暂不实现，作为后续迭代的方向。

第一，用户画像与跨会话记忆。记录用户的学习偏好、薄弱知识点和历史表现，实现跨会话的个性化学习体验。

第二，多语言代码支持。将代码分析和解读能力从 Python 扩展至 Java、C、C++ 等常用编程语言。

第三，主动学习提醒。根据用户的学习节奏，主动推送复习提醒或每日一题，帮助用户保持学习习惯。

第四，知识点问答增强。引入结构化知识库，针对常见编程概念提供更深入、更系统的解释和对比分析。

3.2 非功能性需求

非功能性需求规定了系统的质量属性和约束条件，是保证用户体验和项目质量的基础。

第一，API 封装代码占比控制在总代码量的百分之二十以内。核心价值应体现在提示词设计、业务逻辑和交互体验上，而非简单地调用第三方接口。

第二，提示词设计方案需在项目文档中详细说明。作为本项目的核心设计内容，提示词的层次结构、场景适配策略和质量保障措施均需完整呈现。

第三，需设计至少五个代表性测试场景，覆盖错误解读、解题引导、知识点问答、激励话术和上下文理解等各项核心能力，并提供明确的评价维度。

3.3 设计约束

项目在技术和实现层面面临以下约束条件。

第一，使用 DeepSeek 作为大语言模型服务提供商。其 API 与 OpenAI 接口兼容，可直接使用标准 openai Python SDK，降低接入复杂度。注册即提供免费额度，适合学生项目场景。

第二，采用 Flask 加原生前端技术栈构建 Web 界面。不依赖 Gradio 或 Streamlit 等快速原型框架，以获取完全的界面定制能力，满足专用代码输入区、自定义聊天气泡等复杂 UI 需求。

第三，数据存储采用内存字典加 JSON 文件的轻量方案。不使用 SQLite 或 Redis 等外部数据库，实现零环境配置部署。JSON 文件可直接打开查看和编辑，便于演示和作业提交。

第四，消息分类采用规则匹配而非调用大模型。通过预定义关键词库实现零延迟、零成本的意图识别，过程可解释、规则可调试，适合教学演示场景。
