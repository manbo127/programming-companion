# 程序设计学习智能学伴“小码”项目分析与设计

## 1. 设计概述

“小码”采用浏览器/服务器架构。浏览器只负责交互和展示；Flask 负责身份边界、业务编排、学习数据和模型调用；SQLite 保存全部业务数据；DeepSeek 负责生成教学语言。系统的核心不是简单转发模型，而是在调用前后加入分类、诊断、检索、画像、教学脚手架、学习事件和安全边界。

## 2. 总体架构

```text
Browser
  ├─ 首页与聊天界面
  ├─ 历史会话、学习画像、学习数据
  └─ 账号、诊断与资料引用
        │ HTTPS / JSON
Nginx（TLS、静态资源、限流、反向代理）
        │ 127.0.0.1:8000
Gunicorn gthread
        │
Flask Application Factory
  ├─ API：认证、会话、消息、画像、学习、提醒、健康
  ├─ Service：上下文、画像、诊断、复习、分析、激励
  ├─ Repository：资源所有权与持久化操作
  ├─ Knowledge：确定性检索与官方来源
  └─ LLM Gateway：DeepSeek / FakeLLM
        ├─ SQLite + Alembic
        └─ DeepSeek API
```

### 2.1 分层职责

| 层 | 主要职责 | 代表模块 |
|---|---|---|
| API | 输入校验、身份获取、HTTP 状态和统一响应 | `companion/api` |
| Service | 场景分类、业务编排、分析和教学策略 | `companion/services` |
| Repository | 查询、创建、去重和所有权条件 | `companion/repositories` |
| Model | 关系、约束、索引和数据字段 | `companion/models` |
| LLM | 供应商适配、重试、超时和遥测 | `companion/llm` |
| Knowledge | 审核知识条目、评分检索和来源输出 | `companion/knowledge` |

## 3. 核心业务流程

### 3.1 消息处理

1. 从签名 HttpOnly Cookie 获取 client_id，并验证会话所有权。
2. 使用 client_message_id 执行幂等检查，避免重复提交生成两次回答。
3. 保存 pending 用户消息，识别场景、语言、错误和情绪。
4. 提取学习主题，检索本地知识库，构造结构化诊断和教学阶段。
5. 读取手动画像与安全跨对话摘要，组装分层 system prompt。
6. 按 Token 预算加入最近上下文；必要时加入较早对话摘要。
7. 先提交用户消息，再在数据库事务外调用 DeepSeek。
8. 保存回答、来源、诊断、模型遥测和学习事件。
9. 更新复习计划、主动提醒、会话摘要和跨对话画像。

### 3.2 教学脚手架

- 报错场景：错误含义 → 证据位置 → 可能原因 → 一项检查 → 验证问题。
- 解题场景：理解题意 → 拆分问题 → 形成算法 → 验证反思。
- 知识场景：一句话定义 → 类比 → 小片段 → 对比 → 延伸问题。
- 自由交流：保持自然回应，不强行套用教学模板。

### 3.3 知识检索

知识库保存人工整理的短摘要、语言、主题、关键词和官方链接。检索器按关键词、主题和语言确定性打分，最多返回三个条目。模型只接收本地摘要，并被明确告知资料是只读参考；浏览器显示持久化后的来源链接。

## 4. 数据库设计

### 4.1 clients

| 字段 | 类型 | 约束/用途 |
|---|---|---|
| id | String(36) | UUID 主键 |
| email | String(254) | 可空、唯一、索引 |
| password_hash | String(255) | 仅保存哈希 |
| is_anonymous | Boolean | 匿名/注册身份 |
| failed_login_attempts | Integer | 登录失败计数 |
| locked_until | DateTime | 临时锁定时间 |
| created_at / last_seen_at / last_login_at | DateTime | 生命周期审计 |

### 4.2 learner_profiles

以 client_id 为主键和外键，保存 nickname、skill_level、preferred_languages、learning_goal、feedback_style、memory_summary、memory_enabled、memory_reset_at 和 memory_updated_at。删除 Client 时级联删除。

### 4.2.1 account_sessions

每个设备登录生成独立随机令牌，数据库只保存 token_hash，并记录 expires_at、last_seen_at 和 revoked_at。退出登录只撤销当前设备会话，其他设备不受影响；旧版仅含 client_id 的 Cookie 不能访问注册账号。

### 4.3 conversations

保存会话标题、滚动摘要、积极/挫败连续计数、最近错误指纹和时间戳；client_id 建索引并作为所有权边界。

### 4.4 messages

保存 role、content、code、error_text、scene、detected_language、error_type、emotion、emotion_score、motivation_text、status、sources_json、diagnosis_json、Token、延迟、模型、请求 ID、尝试次数和结束原因。`conversation_id + sequence_no` 与 `conversation_id + client_message_id` 均为唯一约束。

### 4.5 learning_events

保存主题、语言、错误类型、事件类型和结构化 metadata_json，用于掌握度、趋势、场景与情绪分布分析。

### 4.6 reminders

保存提醒类型、内容、状态和 dedupe_key；`client_id + dedupe_key` 唯一，避免相同提醒重复生成。

### 4.7 review_plans

保存主题、原因、interval_index、next_review_at、last_reviewed_at 和状态；`client_id + topic` 唯一。复习间隔为 1/3/7/14/30 天。

## 5. Prompt 设计

Prompt 由以下部分顺序组合：

1. 版本号与基础人设；
2. 不可信数据边界；
3. 当前场景规则；
4. 激励原则；
5. 用户画像与跨对话摘要；
6. 语言诊断提示；
7. 可核验知识参考；
8. 本轮结构化诊断与教学脚手架；
9. 情绪状态提示。

代码、报错、用户文本、画像摘要和较早对话摘要都被标记为数据，不得覆盖系统行为。报错和解题场景明确限制完整答案输出。

## 6. DeepSeek 接入设计

- 使用 OpenAI 兼容 SDK，默认模型 `deepseek-v4-flash`。
- 可配置 thinking 与 reasoning_effort，当前教学场景默认关闭思考模式以降低等待时间。
- 单次超时和整个调用时间预算分离。
- 仅对 408、409、429、5xx、超时和连接失败退避重试。
- 401/403 和普通 4xx 立即失败并映射为安全错误码。
- 空回答视为无效响应；调用记录保存 model、request_id、attempts、finish_reason 和 Token。
- 测试使用 FakeLLM，不消耗真实额度。

## 7. 身份与安全设计

- 匿名用户首次请求时生成 UUID，并只在签名 Cookie 中保存。
- 注册将当前匿名 Client 原地升级，因此历史不会丢失。
- 登录只通过邮箱和密码查找账号，不接受前端提交 client_id。
- 密码使用 Werkzeug 安全哈希；连续五次失败锁定十五分钟。
- 所有会话、画像、学习数据和提醒查询都带 client_id 条件。
- 生产启用 Secure/HttpOnly/SameSite Cookie、CSRF、HSTS、CSP、Frame Deny 和 MIME Sniffing 防护。

## 8. 数据与部署设计

SQLite 开启 foreign_keys、WAL、busy_timeout 和 synchronous=NORMAL。迁移文件保持单一 Alembic head。`backup-database` 使用 SQLite Online Backup API 并执行 integrity_check。

真实部署为阿里云香港 ECS：Nginx 提供 TLS 和限流，Gunicorn 只监听本机，Systemd 在启动前执行数据库迁移并限制服务权限。SQLite 模式固定单 worker 多线程；未来切换 PostgreSQL 后可使用连接池和多个 worker。

Vercel 不是当前生产架构。Python Function 的临时文件系统无法承担本项目 SQLite 的长期持久化，因此旧 Vercel 入口已移除，避免文档与实际运行不一致。

## 9. 测试与运维

- 单元测试：分类、语言、错误解析、情绪、上下文、检索、诊断、DeepSeek 重试和数据库配置。
- 集成测试：API、幂等、跨会话画像、复习计划、引用持久化、账号隔离、备份和指标。
- 生产安全测试：CSRF、反向代理、HTTPS Cookie 和安全响应头。
- CI：Python 3.11/3.12 测试、全部前端 JS 语法检查、Alembic 单 head 检查。
- 运维：JSON 请求日志、request_id、`/health`、`/ready`、`/metrics` 和只读冒烟脚本。
