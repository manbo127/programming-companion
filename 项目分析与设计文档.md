# 程序设计学习智能学伴 — 项目分析与设计文档

---

## 一、需求分析

### 1.1 项目背景

在程序设计学习过程中，学习者常常面临以下困境：

| 困境 | 具体表现 |
|------|---------|
| 遇到问题无人可问 | 写代码报错后不知如何排查，周围缺乏能解答的人 |
| 错误信息难以理解 | 编译器和运行时的报错文本对初学者来说晦涩难懂 |
| 缺乏持续学习动力 | 反复出错后容易产生挫败感，甚至放弃学习 |
| 缺乏系统引导 | 想知道解题思路而非直接看答案，但网上资源大多是现成代码 |

### 1.2 功能需求（6项核心功能）

#### 功能1：基础对话能力

- 学伴能与用户进行自然流畅的自由对话
- 自动识别问题主题范围（代码错误/解题思路/知识点疑问/一般交流）
- 角色定位为"耐心温和的编程学习陪伴者"，语气友好鼓励

#### 功能2：代码错误解读能力

- 用户可以提交出错代码 + 系统返回的错误信息
- 学伴理解代码意图，分析错误原因，用通俗语言解释
- **关键约束**：避免直接输出大段正确代码，侧重于解释"为什么错"和"往哪个方向改"

#### 功能3：解题思路引导能力

- 用户询问编程题目时，不得直接给出完整答案代码
- 采用苏格拉底式分步引导：
  1. 先帮助理解输入输出
  2. 提示解决问题的关键步骤
  3. 针对每个步骤提出引导性问题
  4. 根据用户回答决定下一步提示
- 多轮对话逐步推进，直到用户理解解题思路

#### 功能4：激励话术能力

- 感知用户学习状态，给予情感化反馈
- 识别两种典型场景：
  - 连续正确/积极进展 → 肯定和表扬
  - 连续错误/挫败情绪 → 鼓励和安慰
- 话术自然真诚，避免机械重复

#### 功能5：交互界面

- Web 聊天界面形式
- 基本元素：对话消息展示区、用户文本输入框、发送按钮
- 专用区域：代码输入区、错误信息输入区
- 支持在统一输入框中按约定格式提交代码和错误信息

#### 功能6：对话上下文记忆能力

- 记住本次对话过程中的交流历史
- 在后续回答中引用历史信息
- 理解代词指代（如"那列表和元组有什么区别"中的"那"）

### 1.3 扩展功能（4项，本次暂未实现）

- 功能7：用户画像与跨会话记忆
- 功能8：多语言代码支持（Python/Java/C/C++）
- 功能9：主动学习提醒
- 功能10：知识点问答增强

### 1.4 非功能需求

- API 封装代码不超过总代码量的 20%
- 提示词设计需在报告中详细说明
- 需要设计至少 5 个测试场景覆盖各项核心能力

## 二、技术选型

### 2.1 大模型 API 选型：DeepSeek

| 对比维度 | DeepSeek | 通义千问 | 文心一言 | 智谱AI |
|---------|----------|---------|---------|--------|
| 响应速度 | 快（~2-5秒） | 快 | 中等 | 中等 |
| 免费额度 | 新用户赠送 tokens | 有免费额度 | 有免费额度 | 有免费额度 |
| 中文理解 | 优秀 | 优秀 | 优秀 | 优秀 |
| 接口兼容性 | OpenAI 兼容（直接用 openai SDK） | 需专用 SDK | 需专用 SDK | OpenAI 兼容 |
| 注册便捷度 | 简单（手机号注册） | 需阿里云账号 | 需百度账号 | 需手机号 |

**选择理由**：DeepSeek 的 OpenAI 兼容接口使代码最简洁，可直接使用标准的 `openai` Python SDK，无需学习新的 API 调用方式。中文理解能力强，适合编程教学场景。

### 2.2 界面技术选型：Flask + HTML/CSS/JS

| 对比维度 | Flask + HTML | Gradio | Streamlit | 命令行 |
|---------|-------------|--------|-----------|--------|
| 开发效率 | 中等 | 高 | 高 | 低 |
| 界面美观度 | 完全可控 | 较固定 | 较固定 | 无 GUI |
| 代码块展示 | 完美支持 | 一般 | 一般 | 不支持 |
| 多轮对话展示 | 完美支持 | 一般 | 一般 | 文本形式 |
| 定制化能力 | 高 | 低 | 低 | 无 |
| 学习成本 | 中等 | 低 | 低 | 低 |

**选择理由**：本次项目需要专用代码输入区、错误信息输入区、自定义聊天气泡等复杂 UI 元素，Gradio 和 Streamlit 虽然开发快但定制能力差。Flask + 原生前端可以完全控制界面细节，适合展示和演示。

### 2.3 数据存储选型：内存字典 + JSON 文件

| 对比维度 | 内存+JSON | SQLite | Redis |
|---------|----------|--------|-------|
| 实现复杂度 | 低 | 中 | 中高 |
| 读写效率 | 高（内存） | 中 | 高 |
| 跨会话持久化 | 支持 | 支持 | 需配置持久化 |
| 无需环境配置 | 是 | 是 | 需安装 Redis |
| 数据可读性 | 高（JSON直接打开） | 需 SQL 查询 | 低 |

**选择理由**：JSON 文件零配置、可直接查看、方便演示和提交作业。课程设计场景下数据量小，JSON 的性能完全足够。

```

│ 用户在界面填写: 消息文本 + 代码 + 错误信息                    │

┌─────────────────────────────────────────────────────────┐

└──────────────────────┬──────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────┐
```

### 4.3 情绪检测与激励触发规则

```
用户消息 → 关键词扫描
│
├─ 包含 "明白了" "原来如此" "成功了" → is_positive = true
│   └─ consecutive_success += 1
│       └─ ≥ 3 次 → 触发表扬话术
│       └─ = 1 次 → 简短肯定
│
├─ 包含 "好难" "又错了" "不懂" "崩溃" → is_frustrated = true
│   └─ consecutive_errors += 1
│       └─ ≥ 2 次 → 触发鼓励话术 + 检查是否需要主动帮助
│       └─ = 1 次 → 安慰话术
│
└─ 包含 "?" "什么意思" "能再讲一遍吗" → is_confused = true
    └─ 切换解释方式，不改变连续计数
```

## 五、提示词设计

### 5.1 设计思路

提示词工程是本项目的核心。提示词分为**基础人设层**和**场景约束层**：

```
┌──────────────────────────────────────────────┐
│            基础人设 (Base System Prompt)       │
│   - 角色: 耐心温和的编程学习陪伴者"小码"         │
│   - 性格: 耐心/鼓励/通俗/真诚/自然              │
│   - 行为准则: 不直接给答案/侧重理解/灵活应对      │
├──────────────────────────────────────────────┤
│          场景约束层 (Scene-specific Rules)     │
│                                              │
│  ┌────────┐ ┌──────────┐ ┌────────┐          │
│  │ 错误解读│ │ 解题引导  │ │知识点问答│          │
│  │ 解释原因│ │ 分步提问  │ │分层解释 │          │
│  │ 不给答案│ │ 不给代码  │ │使用比喻 │          │
│  │ 给方向  │ │ 每次一步  │ │示例说明 │          │
│  └────────┘ └──────────┘ └────────┘          │
├──────────────────────────────────────────────┤
│          激励指导层 (Motivation Guide)         │
│   - 表扬: 具体而非笼统                         │
│   - 鼓励: 共情 + 正常化困难 + 成长型思维        │
│   - 原则: 自然真诚、避免机械重复                │
└──────────────────────────────────────────────┘
```

### 5.2 各场景提示词的核心约束

**错误解读模式**：
```
1. 先共情 → 2. 定位错误 → 3. 解释原因（用比喻）→ 4. 给出方向（不写完整代码）→ 5. 预防建议
❌ 不要输出完整修复后的代码
✅ 要说"你可以试试把第X行改成..."或"考虑用...来替代..."
```

**解题引导模式**：
```
1. 理解问题 → 2. 分析输入输出 → 3. 分解步骤 → 4. 引导性提问 → 5. 根据回答推进
❌ 绝对不要直接给出完整的代码答案
❌ 不要一次性讲完所有步骤
✅ 每次回复只引导一个步骤，用提问推动思考
```

**知识点问答模式**：
```
1. 一句话定义 → 2. 生活化例子 → 3. 简单代码片段 → 4. 对比辨析 → 5. 延伸提问
```

## 八、测试场景设计

### 5个代表性测试场景

| 编号 | 场景类型 | 用户输入 | 期望行为 |
|------|---------|---------|---------|
| 1 | 语法错误解读 | Python 代码 + `IndexError: list index out of range` | 解释索引从0开始、列表长度概念，不给出完整正确代码 |
| 2 | 逻辑错误解读 | 无限循环代码 + 描述"程序一直运行不停止" | 解释循环条件逻辑、终止条件的重要性 |
| 3 | 解题思路引导 | "如何用 Python 判断一个数是否是素数？" | 分步引导：输入→判断逻辑→特殊情况→优化，不直接给代码 |
| 4 | 知识点问答 | "什么是递归？能举个例子吗？" | 用套娃/镜子比喻解释，给出简单代码片段说明 |
| 5 | 激励话术 | 连续2次提交相同错误 + "我又遇到这个错误了，好难" | 检测挫败情绪，给予鼓励话术，主动提议讲解相关概念 |

### 评价维度

- **准确性**：回复内容在技术上是否正确
- **引导性**：是否引导用户思考而非直接给答案
- **语气友好度**：是否符合"耐心温和"的角色定位
- **上下文感知**：多轮对话中是否正确引用历史信息

## 九、关键设计决策总结

| 决策 | 选择 | 核心理由 |
|------|------|---------|
| 消息分类 | 规则匹配，非 LLM | 零延迟、零成本、可解释、可调试 |
| 激励话术 | 自定义代码 | 质量可控、避免 LLM 生成机械重复内容 |
| 不同场景不同温度 | error=0.3, guidance=0.6, general=0.7 | 错误分析需准确性，引导需创造性，一般对话需自然 |
| 前端 Markdown 渲染 | 纯 JS 正则 | 零依赖、轻量、足够满足需求 |
| JSON 而非 SQLite | JSON 文件 | 可直接查看编辑、零配置、便于演示提交 |
| 客户端会话ID | localStorage + UUID | 跨页面刷新保持会话，无需服务端 Cookie |
| 独立代码/错误输入区 | 分离的 textarea | 引导用户正确使用，降低格式错误概率 |
| 语言检测 | 正则特征匹配 | 覆盖 Python/Java/C/C++ 中英文关键词 |
| 提示词设计 | 基础层 + 场景层 | 复用公共人设，各场景独立约束 |
